## Custom Project

### Dataset

This project used a custom online course enrollment dataset named:

```text
enrollments_sowers.csv
```

The dataset contained enrollment transaction records for online learning
courses such as artificial intelligence, data science, cloud computing,
web development, and cybersecurity.

Each enrollment record included fields such as:

- order_id
- datetime
- region_id
- currency_code
- product_id
- unit_price
- quantity
- customer_id
- payment_method
- device_type
- referral_source
- discount_code
- customer_note

This project modified the original sales dataset into a new educational
enrollment analytics dataset.

Reference datasets used for validation and enrichment included:

- `products_sowers.csv`
- `regions.csv`
- `currencies.csv`
- `discount_codes.csv`

---

### Data Contract

The enrollment data contract defined validation rules for all Kafka
messages.

Required fields included:

- order_id
- datetime
- region_id
- product_id
- unit_price
- quantity
- payment_method

Optional fields included:

- referral_source
- discount_code
- customer_note

Validation rules checked:

- valid region IDs
- valid product IDs
- allowed payment methods
- allowed device types
- allowed referral sources

A message became invalid if:

- required fields were missing
- invalid reference values were used
- unsupported payment methods or regions appeared

---

### Kafka Messages

The Kafka producer streamed enrollment transaction messages through
Kafka.

The project used the Kafka topic:

```text
enrollments-sowers
```

The Kafka message key used:

- `region_id`

The producer sent enrollment records containing:

- customer enrollment details
- course purchase information
- regional information
- payment and discount details

The message fields were expanded from the original sales example to
support educational enrollment analytics.

---

### Consumer Validation

The Kafka consumer validated incoming messages before processing them.

Validation checks included:

- required field checks
- allowed payment methods
- valid region IDs
- valid product IDs
- allowed device types
- allowed referral sources

When messages passed validation:

- they were enriched
- processed totals were calculated
- results were written to the output CSV

When messages failed validation:

- they were rejected
- validation errors were logged
- rejected messages were written to:

```text
data/output/rejected_enrollments_sowers.csv
```

Validation protected the analytics workflow from invalid data entering
the processed dataset.

---

### Data Engineering and Enrichment

The consumer enriched incoming enrollment messages with additional
calculated fields.

Derived fields included:

- subtotal
- tax_amount
- total
- discount_applied
- processed_timestamp

Reference data from `regions.csv` was used to calculate regional taxes.

The project transformed raw enrollment records into processed analytics
records with Kafka metadata and calculated totals.

Additional calculations were added beyond the original example,
including:

- discount tracking
- processed timestamp tracking

---

### Streaming Analytics

The consumer generated running analytics summaries while consuming Kafka
messages.

Streaming summaries included:

- total enrollment revenue
- average enrollment value
- minimum enrollment value
- maximum enrollment value

The running statistics updated dynamically as enrollment messages were
consumed from Kafka.

### Interpretation

This project demonstrated how Kafka streaming pipelines can be adapted
to different business problems while maintaining validation and
analytics workflows.

Compared to the original example:

- the dataset changed from retail sales to educational enrollments
- new validation rules were introduced
- additional analytics fields were added
- a separate Kafka topic was created

The project showed the importance of validating streaming messages
before analytics processing.

The enrichment workflow demonstrated how raw Kafka events can be
transformed into more useful business analytics records.

The running summaries provided insights into:

- enrollment revenue
- customer enrollment activity
- regional enrollment performance
- course purchase behavior

The validated and enriched streaming analytics pipeline demonstrated
how organizations can monitor real-time enrollment activity while
maintaining data quality controls.
